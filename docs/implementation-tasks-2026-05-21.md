### 13. 自動投稿時のファクトチェック裏側実行

- [ ] ファクトチェック失敗時の扱いを決める

  - 警告付きで下書き保存する
  - ログに残して続行する
- [ ] Perplexity APIキー未設定時の扱いを決める
  - 公開を止めて下書き保存する
  - 管理者・担当者へ通知して設定完了まで保留する
  - 暫定的にログのみ残して続行する
- [ ] 自動投稿ログにファクトチェック結果を残す
- [ ] ユーザーに不要な設定項目は出さない

検討メモ:

- ファクトチェックを「公開前の安全工程」として扱うなら、APIキー未設定は通常の警告ではなく設定不備として扱う必要がある。
- 推奨案は「自動投稿では公開せず下書き保存し、ChatWork通知または実行履歴で設定不備を明示する」。公開を続行すると、必須工程を通らない記事が外部公開されるためリスクが高い。
- ただし初期導入や検証環境ではAPIキー未設定でも記事生成だけ確認したいケースがあるため、環境または管理者設定で一時的に「ログのみで続行」を許可する余地を残す。
- 失敗・未設定・重大指摘は別の状態としてログに残す。単に「ファクトチェック失敗」とまとめると、設定不備と外部API障害と記事内容の問題を切り分けにくい。

完了条件:

- 自動投稿でも裏側でファクトチェックが実行される
- 失敗時の挙動が明確でログから追える
- APIキー未設定時に、公開可否・通知・ログの扱いが明確になっている

## 作業メモ 2026-05-26

### WordPressカスタム投稿タイプ `news` のカテゴリー付与

- WordPress接続設定:
  - デフォルトカテゴリー: `column`
  - 投稿タイプ: `news`
- WordPress管理画面では、対象taxonomyは `news_category`。
  - 管理画面URL例: `/wp/wp-admin/edit-tags.php?taxonomy=news_category&post_type=news`
  - 「コラム」のスラッグは `column`
- 記事一覧ページからの投稿経路は、予約投稿ではなくフロント側。
  - `ArticlesList.tsx` → `useWordPressPublish.ts` → `src/services/wordPressService.ts`
- REST API確認結果:
  - `/wp-json/wp/v2/news` への投稿は成功している
  - `/wp-json/wp/v2/news_category?slug=column` は404
  - `news_category` がREST APIに公開されていないため、REST経由では「お知らせカテゴリー」にチェックを付けられない
- XML-RPC fallback確認結果:
  - Supabase Edge Function `wp-taxonomy-assign` 経由で `wp.editPost` を試した
  - WordPress側が `xmlrpc.php` を403で拒否
  - Wordfenceまたはサーバー/WAFでXML-RPCがブロックされている可能性が高い
- 現時点の結論:
  - アプリ側だけで完結させるには、RESTで `news_category` が見えるか、XML-RPCが通る必要がある
  - 現状はどちらもWordPress側で拒否されている
- 推奨対応:
  - XML-RPCを開けるより、WordPress側で `news_category` の `show_in_rest` を有効化する方が安全
  - 既存の `register_taxonomy('news_category', ...)` に以下を追加する想定:

```php
'show_in_rest' => true,
'rest_base' => 'news_category',
```

- 次回確認すること:
  - Wordfenceの Live Traffic / Firewall / Blocking で `xmlrpc.php` の403理由を確認
  - テーマまたはカスタム投稿タイプ定義箇所で `news_category` の `show_in_rest` を有効化できるか確認
  - `https://zuiryo-seiki.com/wp/wp-json/wp/v2/news_category?slug=column` がJSONを返す状態にできるか確認
