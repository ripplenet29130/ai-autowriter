# AutomaticWriter クライアント固有設定マトリクス

このドキュメントは、Phase 2 の基準資料です。  
外販運用に向けて、`共通機能` と `クライアント差分` を分け、どこで設定するかを固定します。

---

## 1. このドキュメントの目的

1. 何を共通 package / 親アプリに残すかを明確にする
2. 何を導入クライアントごとの差分として持つかを明確にする
3. `.env`、DB、app ファイルのどこで管理するかを固定する
4. `client-template` を作る前提を揃える

---

## 2. 用語定義

- 導入クライアント: AutomaticWriter を契約・利用する HP 業者 / Web 制作会社 / Web 運用会社
- 投稿先サイト: 導入クライアントが管理する顧客の WordPress サイト
- クライアント差分: 原則として導入クライアント単位で分ける設定
- サイト差分: 投稿先サイトごとに複数登録・切り替えする設定

---

## 3. 現時点の前提

- 親アプリは `internal`、子アプリは `client` を基本方針とする
- `VITE_DEPLOYMENT_MODE` により `internal / client` を切り替える実装は入っている
- client モードではファクトチェックは常時有効扱い
- 外販先は、1つの WordPress サイトを持つ一般企業ではなく、複数の顧客サイトを管理する HP 業者を主対象とする
- ロゴ、アプリ名、ブランドカラーは導入クライアントごとに変えず、共通表示のまま進める
- URL / ドメインは導入クライアントごとに Vercel へ個別アップロードして設定する
- Supabase Project は導入クライアントごとに分ける
- WordPress 接続先は、導入クライアントの DB 内に投稿先サイト単位で複数登録できる前提にする
- ただし、アプリ単位の完全分離はまだ未実装
- そのため、現時点では「将来分離する前提で差分を整理する」段階

---

## 4. 共通で持つもの

以下は、原則として共通 package または親アプリ側に残す。

- 記事生成フロー
- 対話型記事生成
- 見出し編集機能
- ファクトチェックの実行ロジック
- WordPress 投稿ロジック
- スケジューラー機能
- AI 設定管理
- 記事一覧
- 共通 UI コンポーネント
- エラーハンドリング
- アプリ名
- ロゴ
- ブランドカラー

## 5. 導入クライアント差分として持つもの

以下は、導入クライアントごとに変更できる、または今後判断する前提で整理する。

- ドメイン
- `VITE_DEPLOYMENT_MODE`
- `VITE_ACCESS_PASSWORD` の有無
- Supabase 接続先
- Chatwork 通知先
- メニュー表示方針
- 初期設定値

## 6. サイト差分として持つもの

以下は、導入クライアントの中で投稿先サイトごとに複数登録・切り替えできる前提で整理する。

## 7. 設定マトリクス

| 項目 | 基本方針 | 現状 | 設定場所 | ステータス | 備考 |
|---|---|---|---|---|---|
| アプリ名 | 共通 | 共通表示 | app ファイル | 方針決定 | 導入クライアントごとには変えない |
| ロゴ | 共通 | 共通表示 | app ファイル / assets | 方針決定 | 導入クライアントごとには変えない |
| favicon | 応相談 | 未分離 | app ファイル / public | 未決定 | ロゴ共通なら原則共通でもよい |
| ブランドカラー | 共通 | 共通表示 | app ファイル / CSS | 方針決定 | 導入クライアントごとには変えない |
| ドメイン | 導入クライアント差分 | Vercel で個別設置予定 | Vercel | 方針決定 | 導入クライアントごとに Vercel へアップロードして設定 |
| `VITE_DEPLOYMENT_MODE` | 親子で差分 | 実装済み | `.env` / Vercel env | 実装済み | `internal` 以外は `client` 扱い |
| `VITE_ACCESS_PASSWORD` | 導入クライアント差分候補 | 未実装 | `.env` / Vercel env | 未実装 | パスワードゲート自体はまだ未実装 |
| Supabase URL | 導入クライアント差分 | 既存 env 対応あり | `.env` / Vercel env | 方針決定 | 当社管理の Supabase Project を導入クライアントごとに作成して設定 |
| Supabase anon key | 導入クライアント差分 | 既存 env 対応あり | `.env` / Vercel env | 方針決定 | URL とセットで当社が導入クライアントごとに管理 |
| Supabase DB 方針 | 導入クライアント差分 | 導入クライアントごとに分離 | 運用方針 | 方針決定 | 当社側の Supabase Organization で導入クライアントごとに Project を分け、当社が初期設定・保守・請求管理を行う |
| WordPress URL | サイト差分 | 複数設定可能 | DB | 実装済み | 投稿先サイトごとに登録・切り替えする |
| WordPress ユーザー名 | サイト差分 | 複数設定可能 | DB | 実装済み | 投稿先サイトごとの投稿用ユーザー |
| WordPress アプリケーションパスワード | サイト差分 | 複数設定可能 | DB | 実装済み | 投稿先サイトごとに発行・保存する |
| Chatwork API Token | 導入クライアント差分 | 設定可能 | DB (`app_settings`) | 実装済み | env ではなく DB 保存 |
| Chatwork 通知先ルーム ID | 導入クライアント差分 / サイト差分 | 設定可能 | DB | 実装済み | 導入クライアント共通通知先、または案件・サイト別通知先として扱う |
| ファクトチェック有効化 | 親子で差分 | 実装済み | DB + deployment mode | 実装済み | client では常時有効扱い |
| 自動修正有効化 | 導入クライアント差分 | 設定可能 | DB | 実装済み | client でも ON/OFF 自体は残る |
| 記事生成後の自動ファクトチェック | 親子で差分 | 実装済み | deployment mode | 実装済み | client モード時に自動実行 |
| メニュー表示 | 導入クライアント差分候補 | 未分離 | app ファイル | 未実装 | 将来的な機能制限に使う |
| 非表示機能の制御 | 導入クライアント差分候補 | 未分離 | app ファイル | 未実装 | 単なる env 切り替えでは不足 |
| 初期設定文言 | 導入クライアント差分候補 | 未分離 | app ファイル | 未実装 | UI 整理フェーズで対応 |
| 利用マニュアル URL | 導入クライアント差分候補 | 未分離 | app ファイル / docs | 未実装 | 納品物とセットで整理 |

## 8. `.env` に寄せるもの

以下は、原則として `.env` または Vercel Environment Variables に寄せる。

- `VITE_DEPLOYMENT_MODE`
- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`
- `VITE_ACCESS_PASSWORD`（将来実装時）

方針:

- 親アプリは `VITE_DEPLOYMENT_MODE=internal`
- 子アプリは `VITE_DEPLOYMENT_MODE=client`
- 導入クライアントごとに Vercel 側 env を分ける

---

## 9. DB に寄せるもの

以下は、運用中に変更されうるため DB 管理を維持する。

- 投稿先サイトごとの WordPress 接続先
- 投稿先サイトごとの WordPress 認証情報
- Chatwork API Token
- Chatwork 通知先
- ファクトチェック詳細設定
- 自動修正 ON/OFF
- スケジュール設定

注意:

- Supabase Project は導入クライアントごとに分離する
- Supabase アカウント / Organization / 請求は当社側で管理する
- 当社が初期設定、Edge Functions、DB設定、環境変数連携、保守を行う
- Supabase 利用料は販売価格または保守費に含め、導入クライアントへ別請求しない

---

## 10. app ファイルに寄せるもの

以下は、クライアント専用 app 側に持たせる想定。

- favicon（差し替える場合）
- メニュー構成
- 表示文言
- 機能の見せ方

## 11. Phase 2 で未決定の論点

- favicon を共通にするか、導入クライアントごとに差し替え可能にするか
- パスワードゲートを本当に入れるか
- メニュー制限を deployment mode だけで行うか、app 分離前提にするか
- Chatwork をクライアント標準機能にするか、オプションにするか
- Chatwork 通知先を導入クライアント共通にするか、投稿先サイトごとにも持たせるか
- 投稿先サイトの登録上限やプラン別制限を設けるか
- 自動修正を client のデフォルト ON にするか

---

## 12. 現時点の結論

- 今すぐ app 分離しなくても、差分定義は先に進められる
- ロゴ、アプリ名、ブランドカラーは共通扱いで進める
- URL / ドメインは導入クライアントごとの Vercel 個別プロジェクトまたは個別デプロイで分ける
- Supabase Project は導入クライアントごとに分ける
- WordPress 接続先は、導入クライアント内の投稿先サイトごとに複数登録・切り替えできる前提にする
- ただし、本番外販運用は env 切り替えだけでは危ない
- `apps/master` と `client-template` に分ける前提で差分項目を固定するのが妥当
- Phase 2 の次の着手は、未決定項目のうち `表示/非表示機能`、`通知先の粒度`、`投稿先サイト数の扱い` を埋めること
