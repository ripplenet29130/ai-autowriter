# AutomaticWriter client側UI・メニュー再設計方針

## 目的

`14_Admin_Client_Implementation_Tasks.md` を前提に、client側のトップ画面とメニュー構成を整理する。

外部提供先のclientが迷わず使えるように、主要機能を前面に出し、未完成または運用上まだ見せない機能は詳細オプションまたは非表示にする。

## 基本方針

- client側の左メニューは機能を細かく分けすぎない
- 「記事作成」は1つのメニューにまとめ、開始方法は画面内で選ばせる
- admin向け、開発者向け、移行作業向けの機能はclient画面に出さない
- 不十分な機能は削除せず、feature flagや詳細オプションで将来利用できる余地を残す
- client画面では専門的なAPI名をなるべく前面に出さず、用途が分かる名称にする

## client側メニュー案

### 1. ホーム

clientがログイン後に最初に見る画面。

表示したい情報:

- 記事数
- 本日の投稿予定
- 有効な予約投稿数
- WordPress接続状態
- AI API設定状態
- キーワード検索機能の接続状態
- WordPress登録上限と現在登録数

主要アクション:

- 記事を作成
- 予約投稿を設定
- WordPressを追加
- AI APIを設定

### 2. 記事作成

左メニューでは「記事作成」1つにまとめる。

記事作成画面内で以下を選択する。

#### 確認しながら作成

現在の対話モードに相当する。

- キーワード検索から開始する
- 競合記事や関連情報を確認する
- タイトル候補を確認する
- 見出しを確認する
- 本文を生成して編集する

UI上の表現は「トレンド分析」ではなく、clientに分かりやすい名称へ変更する。

推奨文言:

- キーワード検索
- 競合記事の確認
- 記事ネタ検索

#### おまかせ作成

現在の自動モードに相当する。

入力方法:

- キーワードから作成
- タイトルから作成
- キーワード + タイトルから作成

本文生成まで一括で進める。

### 3. 記事管理

生成済み記事を管理する画面。

表示・操作:

- 記事一覧
- 下書き
- 公開済み
- 投稿失敗
- 編集
- 再投稿
- 削除

### 4. 予約投稿

スケジューラー機能。

前面に出す項目:

- 投稿先WordPress
- キーワードまたはタイトル
- 投稿頻度
- 投稿時刻
- 公開/下書き
- ChatWork通知

詳細オプションに下げる項目:

- 事実確認（ベータ）

clientには出さない項目:

- 画像生成

### 5. テンプレート

記事作成で使う素材やプリセットをまとめる。

含める機能:

- キーワード
- タイトル
- プロンプト

現在の個別メニューである「キーワード設定」「タイトルリスト設定」は、将来的にはこの配下にまとめる。

### 6. 接続設定

外部サービスや生成AIの接続をまとめる。

含める機能:

- WordPress
- AI API
- キーワード検索設定
- ChatWork通知設定
- 事実確認（ベータ）

## 機能別の表示方針

### キーワード検索機能

残す。

現在「トレンド分析」と呼ばれている機能は、client向けには「キーワード検索」として扱う。

この機能は、記事作成時にキーワードで一度検索し、以下の材料を取得するために必要。

- 競合記事
- 上位記事URL
- 見出し
- 本文抜粋
- 関連情報
- タイトル候補生成の材料

### SerpAPI

残す。

現在の実装では、キーワード検索・競合調査のメインはSerpAPI。

主な流れ:

1. clientがキーワードを入力する
2. `realTrendAnalysisService` が呼ばれる
3. `competitorResearchService` がSupabase Edge Function `competitor-search` を呼ぶ
4. `competitor-search` がSerpAPIでGoogle検索結果を取得する
5. 上位記事を取得し、見出しや本文抜粋を抽出する

client画面上の名称は「SerpAPI」ではなく、基本的には「キーワード検索設定」とする。

ただし、実際の入力欄や管理者向け説明ではSerpAPI名を表示してよい。

### Google Custom Search

client画面では非表示にする。

理由:

- 現在の主要フローではSerpAPIがメイン
- Google Custom Searchは設定項目として残っているが、clientに見せると混乱しやすい
- 将来SerpAPIの代替として使う可能性はあるが、現時点では前面に出さない

扱い:

- コード上はすぐに削除しない
- client向けUIからは外す
- 必要ならadminまたは開発者向け設定として残す

### ChatWork通知

残す。

予約投稿の運用通知として有用なため、client向けにも表示する。

表示場所:

- 予約投稿画面
- 接続設定

推奨文言:

- ChatWork通知
- 投稿完了通知

### 事実確認（ファクトチェック）

削除しないが、前面には出さない。

現時点では機能的に不十分なため、clientの主導線からは下げる。

扱い:

- 左メニューには出さない
- ホームの主要ステータスには出さない
- 記事作成後の編集画面に「事実確認（ベータ）」として表示する
- 予約投稿では詳細オプション内に置く
- 接続設定では必要な場合のみ「事実確認（ベータ）」を表示する
- `feature_flags.fact_check` は残す

推奨文言:

> AIによる補助確認です。公開前に必ず内容をご確認ください。

### 画像生成

client向けUIでは非表示にする。

理由:

- 現時点では機能的に不十分
- 将来的には使う可能性がある
- いまclientに出すと、記事作成や予約投稿の画面が複雑になる

扱い:

- 左メニューに出さない
- ホームのステータスに出さない
- 記事作成画面の画像枚数設定を非表示にする
- 予約投稿画面の画像生成オプションを非表示にする
- 接続設定にも出さない
- `feature_flags.image_generation` は残す
- 新規client作成時のデフォルトはOFF推奨

### ApiKeySetup

client向けUIから外す。

理由:

- 古いAPI設定画面に見える
- 現在の主要導線では使われていない
- トレンド分析用API設定としての説明が残っており、再設計後の名称と合わない

扱い:

- すぐに削除しない
- client導線からは外す
- 必要なら廃止候補として別タスクで整理する

### DataMigration

client向けUIから外す。

理由:

- clientが操作する機能ではない
- データ移行はadminまたは開発者向け作業

扱い:

- clientの設定画面から外す
- 必要ならadminまたは開発者向けページへ移動する

## 表示優先度

### 前面に出す

- ホーム
- 記事作成
- 記事管理
- 予約投稿
- WordPress
- AI API
- キーワード検索設定（SerpAPI）
- ChatWork通知

### 詳細に下げる

- 事実確認（ベータ）

### client画面では非表示

- 画像生成
- Google Custom Search
- DataMigration
- ApiKeySetup
- 開発者向けAPI設定

## 実装時の注意

- 既存のfeature flagは活かす
- 機能を完全削除する前に、まずclient向けUIから外す
- 予約投稿画面は項目が多いため、詳細オプションの折りたたみを検討する
- 「トレンド分析」という表現はclient向けには使わず、「キーワード検索」に寄せる
- APIキー名よりも用途名を優先して表示する
- admin画面では、clientごとの機能ON/OFFを引き続き操作できるようにする

## 実装メモ

- まずはclient向けUIから隠すことを優先し、既存データ構造やバックエンド処理は必要になるまで削除しない。
- 実装完了後、このチェックリストを順に更新する。
- 予約投稿画面は、画像生成UIの非表示と事実確認（ベータ）表記までは完了。主要項目/詳細項目の折りたたみ再構成は次のUI調整として残す。
